# Schedule Management Plan

최종 갱신: 2026-04-20

이 문서는 PhoneShop의 `일정 관리` 메뉴를 신규 도메인으로 추가하기 위한 기준 문서다.
현재 코드에 이미 존재하는 판매, 유지기간 계산, 수납 데이터를 기반으로 `자동 일정`을 계산하고, 운영자가 직접 입력하는 `수동 일정`을 함께 관리하는 것을 목표로 한다.

## 1. 목표

- 유지기간 만료 고객을 달력 기준으로 정기 관리할 수 있게 한다.
- 판매, 수납, 유지기간 만료, 공휴일 정보를 한 화면에서 월 단위로 확인할 수 있게 한다.
- 날짜 클릭만으로 수동 일정을 등록할 수 있게 한다.
- 추후 문자, 카카오톡 알림 기능을 추가할 수 있도록 이벤트 구조를 확장 가능하게 설계한다.

## 2. 메뉴 위치와 권한

- 메뉴 경로: `/schedule`
- 메뉴 라벨: `일정 관리`
- 권한: `ADMIN`, `STAFF` 공용

설정 메뉴가 아니라 운영 메뉴 성격이므로 `/settings/*` 하위가 아닌 워크스페이스 최상위 메뉴로 둔다.

## 3. 핵심 사용자 시나리오

1. 사용자가 월간 캘린더에서 유지기간 만료 예정 고객과 수납/판매 이벤트를 본다.
2. 특정 날짜를 클릭해 신규 수동 일정을 등록한다.
3. 우측 요약 패널에서 현재 연/월의 중요 일정을 빠르게 본다.
4. `펼치기`, `접기`로 요약/상세 보기를 전환한다.
5. 상단 요약 카드에서 오늘 기준 `3일`, `7일` 내 유지만료 대상 수를 확인한다.

## 4. 1차 MVP 범위

포함:
- 월간 캘린더 UI
- 자동 일정 표시
- 수동 일정 등록 / 수정 / 완료 처리 / 삭제
- 우측 요약 패널
- 상단 요약 카드
- no-key 한국 공휴일 API 연동

자동 일정 유형:
- 유지기간 만료 예정
- 판매 / 개통 완료
- 수납 완료
- 공휴일

수동 일정:
- 날짜 클릭 -> 신규 모달
- 제목, 날짜, 고객 연결, 메모, 상태 입력

포함하지 않음:
- 문자 발송
- 카카오톡 발송
- 반복 일정
- 다중 담당자/참석자
- 공휴일 수기 편집

## 5. 자동 일정 설계

### 5-1. 자동 일정은 저장하지 않는다

자동 일정은 DB에 별도 저장하지 않고, 조회 시점에 계산한다.

이유:
- 유지기간 만료일은 판매 이력과 통신사별 개통 가능 규칙이 바뀌면 다시 계산돼야 한다.
- 수납 완료 여부도 실제 잔액/수납 상태 변화에 따라 달라진다.
- 계산형으로 유지하면 중복 데이터 정합성 문제를 줄일 수 있다.

### 5-2. 자동 일정 데이터 소스

- 유지기간 만료 예정:
  - `Sale.saleDate`
  - `Customer.currentCarrierId`
  - `CarrierActivationRule`
  - 계산 함수: `getActivationEligibleDate`
- 판매 / 개통 완료:
  - `Sale.saleDate`
- 수납 완료:
  - `Payment.paymentDate`
  - `Receivable.status = PAID` 또는 개별 수납 기록
- 공휴일:
  - 외부 공휴일 API

### 5-3. 자동 일정 우선순위

기본 우선순위:
1. 공휴일
2. 유지기간 만료 예정
3. 판매 / 개통 완료
4. 수납 완료
5. 수동 일정

우선순위는 날짜 셀 요약과 우측 패널의 기본 정렬 기준으로 사용한다.

## 6. 수동 일정 데이터 설계

신규 모델 이름:
- `ManualSchedule`

예상 필드:
- `id`
- `title`
- `scheduledDate`
- `status`
- `customerId?`
- `saleId?`
- `memo?`
- `createdById`
- `createdAt`
- `updatedAt`

권장 상태값:
- `OPEN`
- `DONE`
- `CANCELED`

1차에서는 반복 일정과 별도 알림 예약 필드는 넣지 않는다.

## 7. 화면 구조

## 7-1. 상단 요약 카드

- 오늘 기준 `3일 내 유지만료`
- 오늘 기준 `7일 내 유지만료`
- 현재 월 자동 일정 수
- 현재 월 수동 일정 수

## 7-2. 본문 레이아웃

- 좌측 메인: 월간 캘린더
- 우측 보조 패널: 현재 조회 연/월 요약

권장 비율:
- 캘린더 `70~80%`
- 우측 패널 `20~30%`

## 7-3. 캘린더 동작

- 월 이동
- 오늘로 이동
- 날짜 셀 클릭 시 신규 일정 모달 오픈
- 날짜별 이벤트 개수/대표 이벤트 표시
- 색상 또는 배지로 이벤트 타입 구분

## 7-4. 우측 요약 패널

기본 상태:
- 중요 일정 요약 5~10건

펼치기 상태:
- 해당 월 전체 이벤트 리스트
- 날짜, 유형, 고객명, 보조 설명 노출

## 8. 공휴일 API 연동 방침

데이터 소스:
- 1차: `https://holidays.hyunbin.page/{year}.json`
- fallback: `https://date.nager.at/api/v3/PublicHolidays/{year}/KR`

연동 방식:
- 서버 측 연 기준 조회 후 월별 필터링
- 연 단위 메모리 캐시
- UI 렌더링 시 직접 외부 API에 의존하지 않도록 설계

권장 환경변수:
- `HOLIDAY_API_BASE_URL`

권장 보조 설정:
- `HOLIDAY_API_FALLBACK_BASE_URL`

장애 대응:
- 1차 공개 소스 실패 시 fallback 공개 API 사용
- fallback도 실패하면 마지막 캐시 데이터 우선 사용
- 캐시도 없으면 공휴일 없이 렌더링하고 내부 로그만 남긴다
- 공휴일 API 장애가 일정 관리 화면 전체를 막지 않도록 한다

## 9. 서버 구현 방향

신규 page-data 계층에서 아래를 합쳐 반환한다.

- 수동 일정 목록
- 유지기간 만료 자동 일정
- 판매 / 개통 완료 자동 일정
- 수납 완료 자동 일정
- 공휴일 목록
- 상단 요약 카드 수치
- 우측 패널용 월간 중요 일정 리스트

권장 파일 예시:

- `src/app/(workspace)/schedule/schedule-page-data.ts`
- `src/components/workspace/schedule-overview.tsx`
- `src/components/workspace/schedule-calendar.tsx`
- `src/components/workspace/schedule-summary-panel.tsx`
- `src/app/actions/schedules.ts`
