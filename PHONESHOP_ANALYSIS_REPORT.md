### 1-2. 요구사항 문서에 잘 정리된 부분

- **데이터 모델 정의**: 고객, 판매, 재고, 수납, 정책의 필드와 규칙이 상세하게 정의됨
- **업무 흐름**: 4가지 핵심 시나리오(신규 판매, 기존 고객 판매, 기기 교체, 미수금 수납)가 단계별로 정리
- **금액 계산 규칙**: 리베이트, 정책 수익, 할인의 개념 분리와 수익 계산 기준이 명확
- **권한 분리**: 관리자/직원 2단계 권한과 메뉴별 접근 제한 정의
- **공통 데이터 규칙**: 식별 기준, 삭제 정책(숨김 처리), 수정 이력, 금액 처리 원칙 등

### 3-2. 스키마 개선 권장 사항

| # | 항목 | 설명 |
|---|------|------|
| 1 | `RatePlan`에 상세 필드 추가 | `voiceMinutes`, `dataGB`, `videoMinutes` 등 요구사항 정의 필드 추가 필요 |
| 2 | `Sale`에 `vatAmount` 필드 활용 | 스키마에는 없으나 요구사항에 부가세 거래별 저장이 명시됨 |
| 3 | 수정 이력(Audit Trail) | 요구사항에 "누가 언제 수정했는지 수정 이력을 남긴다"가 명시되어 있지만 별도 이력 테이블 없음 |

### 4-2. UX 문제점 및 개선 제안

---

#### 🔴 심각도: 높음 (실사용에 즉시 영향)

##### 문제 1: 판매 목록에 필터/검색 기능 없음

**현상**: 판매 관리 페이지는 최근 12건만 표시하며 필터/검색이 없음  
**영향**: 과거 판매 이력을 조회할 수 없어 운영 장부로서의 핵심 가치가 떨어짐  
**개선 방안**:
- 판매일 기간 필터, 고객명 검색, 통신사 필터, 판매 상태 필터 추가
- 전체 판매 목록을 페이지네이션 기반으로 표시
- 필터를 URL 쿼리로 관리하여 새로고침/공유가 가능하도록

##### 문제 2: 페이지네이션 미구현

**현상**: 모든 목록 페이지(재고, 고객, 판매, 미수금)에서 전체 데이터를 한 번에 로드  
**영향**: 데이터가 수백 건 이상 누적되면 페이지 로딩이 느려지고 브라우저 성능 저하  
**개선 방안**:
- 서버 사이드 페이지네이션 도입 (Prisma `skip`/`take`)
- 한 페이지당 20~50건 표시
- 페이지 번호 또는 "더 보기" 버튼 제공

##### 문제 3: 판매 등록 폼에 지원금(subsidyAmount) 입력 UI 누락

**현상**: DB 스키마에는 `subsidyAmount` 필드가 있지만 판매 등록 폼에 입력 UI가 없음  
**영향**: 공시지원금/추가지원금을 기록할 수 없어 매장 수익 분석이 불완전해질 수 있음  
**개선 방안**: 판매 등록 폼의 금액 입력 섹션에 "지원금" 필드 추가

---

#### 🟡 심각도: 보통 (사용성 저하)

##### 문제 4: 콘텐츠 밀도가 낮음

**현상**: 여백이 크고 카드형 UI로 인해 한 화면에 보이는 정보량이 적음  
**영향**: 데이터 중심 업무(재고 확인, 판매 이력 조회)에서 스크롤이 많이 필요  
**개선 방안**:
- 데이터 목록은 컴팩트 테이블 뷰 옵션 제공 (카드 뷰 ↔ 테이블 뷰 전환)
- 메트릭 카드의 높이를 줄이고 한 줄에 4개까지 배치
- `PageIntro` 영역의 크기를 줄이거나 토글 가능하게

##### 문제 5: 사이드바 네비게이션의 모바일 대응 부재

**현상**: `lg:grid-cols-[292px_minmax(0,1fr)]` — 큰 화면에서는 2컬럼, 작은 화면에서는 1컬럼 스택  
**영향**: 모바일/태블릿에서 네비게이션 카드가 본문 위에 모두 펼쳐져 스크롤이 매우 길어짐  
**개선 방안**:
- 모바일에서는 햄버거 메뉴로 사이드바를 숨기고 필요할 때만 열기
- 또는 하단 탭 바(Bottom Tab Bar) 패턴 도입

##### 문제 6: 판매 등록 폼의 고객/재고 선택이 드롭다운만 지원

**현상**: `<select>` 태그로 고객과 재고를 선택하는데, 목록이 많으면 찾기 어려움  
**영향**: 고객이 50명 이상, 재고가 100건 이상이면 드롭다운에서 원하는 항목을 찾는 시간이 길어짐  
**개선 방안**:
- 검색 가능한 Combobox (자동완성 검색 드롭다운) 도입
- 고객은 이름/연락처 검색, 재고는 IMEI/기기명 검색 지원
- 선택 시 상세 정보 미리보기 확장

##### 문제 7: 실시간 유효성 검증 피드백 부족

**현상**: 일부 필드에 유효하지 않은 값을 입력해도 submit까지 가야 에러를 확인 가능  
**영향**: 입력 실수를 빠르게 잡지 못함  
**개선 방안**:
- 필수 필드에 빈 값일 때 즉시 빨간 테두리 + 안내문 표시
- 금액 필드에 음수/문자 입력 시 즉시 경고
- IMEI 중복 검사를 입력 시점에 비동기로 수행

##### 문제 8: 수정 이력 확인 불가

**현상**: 요구사항에 수정 이력 기록이 명시되어 있지만 UI에서 이력 확인 수단이 없음  
**영향**: 언제 누가 데이터를 수정했는지 추적 불가  
**개선 방안**:
- 각 항목의 상세 보기에 "수정 이력" 탭/섹션 추가
- `updatedAt` 외에 별도 `AuditLog` 테이블로 변경 이력 저장

---

#### 🟢 심각도: 낮음 (개선하면 좋음)

##### 문제 9: 빈 상태 / 온보딩 가이드 부족

**현상**: 데이터가 없는 초기 상태에서 다음에 무엇을 해야 하는지 안내가 부족  
**개선 방안**:
- 초기 상태에서 "먼저 기초정보에서 통신사를 등록하세요" 같은 단계별 안내 표시
- 대시보드에 "시작하기" 체크리스트 위젯

##### 문제 10: 키보드 접근성 / 단축키

**현상**: 매장에서 빠른 입력이 중요하지만 키보드 탐색이나 단축키가 없음  
**개선 방안**:
- Tab 순서 최적화
- Enter로 폼 제출
- Ctrl+Enter 같은 단축키로 빠른 등록

##### 문제 11: 다크 모드 / 장시간 사용 배려

**현상**: 따뜻한 톤의 밝은 배경만 지원  
**개선 방안**: 장시간 PC 작업에 대비한 다크 모드 옵션 제공

##### 문제 12: 로딩 상태 표시 부족

**현상**: 서버 액션 처리 중 사용자에게 로딩 피드백이 제한적  
**개선 방안**:
- 전역 로딩 인디케이터 (상단 프로그레스 바)
- 폼 제출 시 버튼 disabled + 스피너 표시 (판매 등록은 이미 구현됨)
