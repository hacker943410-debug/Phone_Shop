## 1. 현재 디자인 평가

### 1-1. 현재 디자인 요약

| 항목 | 현재 값 | 평가 |
|------|---------|------|
| 배경색 | `#f4ede3` (웜 베이지) | 따뜻하지만 장부 도구로는 가벼운 인상 |
| 표면색 | `#fffaf1` (크림) | 배경과 대비가 약해 카드 분리감 부족 |
| 강조색 | `#c7781f` (앰버), `#0f766e` (틸) | 2색 강조는 좋으나 앰버가 경고색과 혼동 |
| 폰트 | Noto Sans KR 400/500/700 | 한글 가독성 양호 |
| 카드 라운딩 | `2rem` ~ `2.25rem` (32~36px) | **과도하게 둥글음** — 모바일 앱 느낌 |
| 그림자 | `shadow-[0_24px_80px_-48px_...]` | 포트폴리오 사이트 수준의 과도한 깊이감 |
| 글래스모피즘 | `bg-white/72 backdrop-blur` | 시각적으로 화려하지만 데이터 가독성 저해 |
| PageIntro 크기 | 제목 `3.15rem`, 패딩 `p-8` | **화면의 30~40%를 차지** — 업무 도구에 부적합 |
| 데이터 목록 | 카드 기반 (인라인 수정 폼 포함) | 한 아이템당 세로 공간이 매우 큼 |

### 1-2. 핵심 문제점

#### ❌ 문제 1: 데이터 밀도가 너무 낮음

| 구간 | 실제 화면 비율 (1440×900 기준) |
|------|------------------------------|
| 사이드바 네비게이션 | ~20% |
| PageIntro (제목/설명) | ~15% |
| MetricCards (요약 카드) | ~20% |
| **실제 데이터 영역** | **~45%** (스크롤 후) |

매장 직원이 하루 종일 사용하는 장부 도구에서는 **데이터 영역이 70% 이상**이어야 합니다.

#### ❌ 문제 2: "디자인 포트폴리오" 느낌

- `rounded-[2.25rem]`(36px) 카드는 드리블(Dribbble) 목업 수준의 둥글기
- 대형 그림자 + 글래스모피즘은 시각적으로 화려하지만 **정보 전달보다 장식에 집중**
- 장부/금융 도구에서는 **"정확하고 신뢰할 수 있다"**는 인상이 더 중요

#### ❌ 문제 3: 재고 목록의 카드 레이아웃

현재 재고 1건당 카드 안에 **전체 수정 폼이 인라인으로 펼쳐져** 있어서:
- 재고 10건만 있어도 스크롤이 매우 길어짐
- 한눈에 목록을 파악하기 어려움
- 새 글 목록 → 상세 보기 → 수정 패턴이 더 효율적

#### ⚠️ 좋은 점 (유지할 것)

- Noto Sans KR 폰트 선택 — 한글 가독성 우수
- 전반적으로 밝은 톤 유지 — 장시간 사용에 적합한 방향
- 색상 시스템 (앰버/틸/로즈/슬레이트) — 상태 구분 체계는 잘 설계됨
- 사이드바 네비게이션 구조 자체 — 메뉴 접근성은 양호

## 2. 디자인 개선 컨셉: "Warm Professional"

### 2-1. 컨셉 설명

> **따뜻하고 밝은 톤은 유지**하되, **장부/금융 도구의 신뢰감과 데이터 효율성**을 결합한 디자인

**레퍼런스 이미지**: Linear, Notion, Stripe Dashboard, 토스 비즈니스 대시보드  
**핵심 키워드**: 밝은 + 정돈된 + 빈틈없는 + 신뢰감

현재 "Bento Portfolio" 스타일을 **"Warm Professional Dashboard"** 스타일로 전환합니다.

```
현재:   눈에 띄는 디자인 → 정보를 찾는 데 시간이 걸림
개선:   정돈된 디자인 → 정보가 바로 눈에 들어옴
```

### 2-2. Before vs After 비교

| 항목 | Before (현재) | After (제안) |
|------|-------------|-------------|
| **배경** | `#f4ede3` 웜 베이지 | `#FAFAF8` 웜 화이트 |
| **카드** | `rounded-[2rem]` + 대형 그림자 + 글래스 | `rounded-xl` (12px) + 1px border + 미세 그림자 |
| **PageIntro** | 제목 `3.15rem` + 설명 여러줄 | 제목 `1.5rem` + 한줄 설명, 높이 50% 감소 |
| **데이터 목록** | 카드 안에 인라인 수정 폼 | 컴팩트 테이블 → 클릭 시 슬라이드 패널 |
| **MetricCard** | `p-5` + `rounded-[1.7rem]` | `p-4` + `rounded-lg` + 한 줄에 4개 |
| **사이드바** | 292px 고정 | 240px + 모바일에서 접히는 드로어 |
| **폼 컨트롤** | `rounded-[1.15rem]` (18px) | `rounded-lg` (8px) |

### 4-2. 사이드바 개선

```
현재: 292px 고정, 카드형 메뉴 아이템 (label + description + badge)
개선: 240px, 심플 리스트 (아이콘 + label), 모바일은 드로어
```

**메뉴 아이템 높이**: 36px → 현재의 ~70px에서 절반으로
**메뉴 간격**: 4px
**활성 메뉴**: 좌측 3px 인디케이터 바 + `bg-accent-subtle`

### 4-3. Page Header 개선

```
현재: PageIntro 컴포넌트 (eyebrow + 3.15rem 제목 + 2줄 설명 + 배지)
     → 높이 약 180~220px

개선: 심플 헤더 (1.5rem 제목 + 우측 액션 버튼)
     → 높이 약 56~64px
```

**변경 이유**: 사용자는 "재고 관리"라는 걸 이미 알고 들어옴. 매번 긴 설명을 읽지 않음.

### 4-4. 데이터 테이블 vs 카드

```
현재: 모든 목록이 카드 기반 → 인라인 수정 폼이 항상 노출
개선: 컴팩트 테이블 → 행 클릭 시 사이드 패널에서 수정
```

| 데이터 건수 | 카드 방식 스크롤 높이 | 테이블 방식 스크롤 높이 |
|-----------|-------------------|---------------------|
| 10건 | ~3000px | ~400px |
| 50건 | ~15000px | ~2000px |
| 100건 | ~30000px | ~4000px |

## 5. 컴포넌트 변경 가이드

### 5-1. MetricCard 개선

**현재**:
```jsx
// rounded-[1.7rem] p-5 그래디언트 배경 + 대형 그림자
<article className="rounded-[1.7rem] border bg-gradient-to-br from-amber-100 to-orange-50 p-5 shadow-[0_18px_45px_-34px_...]">
```

**개선**:
```jsx
// rounded-lg p-4 깔끔한 흰색 + 미세 테두리
<article className="rounded-lg border border-[--color-border] bg-white p-4 shadow-sm">
  <p className="text-caption">판매 가능</p>
  <p className="mt-1 text-2xl font-semibold text-amount tabular-nums">28대</p>
  <p className="mt-1 text-xs text-[--color-text-tertiary]">전체 재고 대비 67%</p>
</article>
```

**배치**: 현재 한 줄에 3개 → **한 줄에 4~5개** (compact)

### 5-2. Panel/Card 개선

**현재**:
```jsx
<section className="rounded-[1.8rem] border bg-white/92 p-5 shadow-[0_18px_50px_-36px_...]">
```

**개선**:
```jsx
<section className="rounded-lg border border-[--color-border] bg-white p-4">
```

### 5-3. 폼 컨트롤 개선

**현재**:
```jsx
<input className="rounded-[1.15rem] border-slate-950/12 px-4 py-3 text-sm" />
```

**개선**:
```jsx
<input className="h-9 rounded-md border border-[--color-border] px-3 text-sm
  focus:border-[--color-accent] focus:ring-2 focus:ring-[--color-accent]/20" />
```

**변경 포인트**:
- 높이 고정 `h-9` (36px) → 일관된 폼 높이
- 라운딩 `rounded-md` (6px) → 비즈니스 도구 표준
- 포커스 링 → 파란색 계열 (신뢰감)

## 6. 페이지별 레이아웃 개선 가이드

### 6-1. 대시보드 (`/`)

```
현재 구조:
[PageIntro 220px] → [필터 패널] → [메트릭 3열] → [운영체크 + 담당자] → [기간흐름 + 최근판매]

개선 구조:
[Header 60px "대시보드" + 기간필터 인라인] → [메트릭 5열 72px] → [2열 패널 그리드]
```

- 기간 필터를 헤더 우측에 인라인 배치 → 별도 패널 불필요
- 메트릭 카드 한 줄에 5개, 컴팩트 높이
- 전체 첫 화면에서 핵심 숫자 + 테이블이 동시에 보이도록

### 6-2. 재고 관리 (`/inventory`)

```
현재 구조:
[PageIntro] → [메트릭 3열] → [등록폼 + 필터 2열] → [카드 목록 (인라인 수정)]

개선 구조:
[Header 60px + 등록 버튼] → [메트릭 바 인라인] → [필터 바] → [테이블 목록]
→ 등록/수정은 슬라이드 오버 패널 또는 모달
```

- 재고 목록은 **컴팩트 테이블**로 전환 (가장 큰 공간 절약)
- 행 클릭 → 우측 슬라이드 패널에서 상세 보기 + 수정
- 등록 버튼 클릭 → 동일 슬라이드 패널에서 등록 폼

### 6-3. 판매 관리 (`/sales`)

```
현재 구조:
[PageIntro] → [메트릭 3열] → [판매 등록 폼 2열] → [최근 목록 12건 카드]

개선 구조:
[Header + 등록 버튼] → [메트릭 바] → [판매 목록 테이블 (필터 포함)]
→ 등록은 별도 페이지 또는 전체 너비 모달
```

- 판매 목록에 **필터와 페이지네이션** 추가 (현재 12건 제한 해소)
- 판매 등록 폼은 정보가 많으므로 별도 페이지 (`/sales/new`)로 분리 가능

### 6-4. 고객 관리 (`/customers`)

```
현재 구조:
[PageIntro] → [메트릭] → [필터 + 목록(왼쪽) + 상세(오른쪽)]

개선 구조:
[Header] → [메트릭 바] → [필터 바] → [목록 테이블(왼쪽) + 상세 패널(오른쪽)]
```

- 현재 마스터-디테일 레이아웃은 좋은 구조 → 유지
- 목록 부분만 카드 → 테이블로 변경

### 6-5. 미수금 관리 (`/receivables`)

- 구조는 현재와 유사하게 유지
- 카드 → 테이블로 변경
- 수납 등록은 행 확장 또는 슬라이드 패널

### 6-6. 기초정보 / 정책 관리 (`/settings/base`, `/settings/policies`)

- 탭 구조 유지 (통신사 / 요금제 / 부가서비스)
- 목록은 테이블로 변경
- 등록/수정은 인라인 폼 또는 모달

## 12. 메뉴 배치 최적화

### 12-1. 현재 메뉴 순서 vs 업무 빈도 기준 재배치

**현재 순서**:
```
1. 대시보드
2. 재고 관리
3. 고객 관리
4. 판매 관리
5. 미수금 관리
── 구분선 (관리자 전용) ──
6. 기초정보
7. 정책 관리
```

**문제**: 실제 매장에서 가장 자주 쓰는 **판매 관리**가 4번째에 있음. 하루에 수십 번 접근하는 메뉴인데 3번 스크롤/클릭을 해야 함.

**제안 순서** (업무 빈도 기반):
```
── 일상 업무 ──
1. 대시보드          ← 출근 시 첫 화면 (오늘 현황 확인)
2. 판매 등록         ← 가장 빈번한 업무 (판매 바로 시작)
3. 판매 내역         ← 판매 후 확인/검색
4. 미수금·수납       ← 판매 직후 또는 고객 방문 시
── 정보 관리 ──
5. 고객 관리         ← 고객 사전 등록 또는 조회
6. 재고 관리         ← 입고 시 또는 재고 확인
── 설정 (관리자) ──
7. 기초정보
8. 정책 관리
```

### 12-2. "판매 등록"과 "판매 내역" 분리

현재는 `/sales` 하나에 등록 폼 + 최근 목록이 같이 있습니다. 이를 분리하면:

| 라우트 | 역할 | 사용 빈도 |
|--------|------|----------|
| `/sales/new` | 판매 등록 (위저드 형식) | 매우 높음 |
| `/sales` | 판매 내역 목록 + 검색/필터 | 높음 |

**이유**: 판매 등록 폼은 매우 복잡한 입력이 필요하므로 목록과 분리하여 **전체 화면을 등록에 집중**하는 것이 오기입 방지에 효과적.

## 13. 업무 흐름 기반 UI 개선 — 오기입 방지 핵심 설계

### 13-1. 원칙: "키보드 타이핑 최소화, 선택·확인 최대화"

```
❌ 현재: 사용자가 값을 직접 타이핑 → 오타, 금액 오류 가능
✅ 개선: 시스템이 후보를 제시 → 사용자가 선택/확인만
```

| 입력 유형 | 현재 방식 | 개선 방식 | 오기입 방지 효과 |
|----------|----------|----------|---------------|
| 고객 선택 | `<select>` 드롭다운 | **검색형 Combobox** (이름/번호 자동완성) | 50명 이상 목록에서 빠르게 찾음 |
| 재고 선택 | `<select>` 드롭다운 | **카드형 선택 UI** (통신사→기기→색상 단계별) | 잘못된 재고 선택 방지 |
| 통신사 선택 | `<select>` 드롭다운 | **칩(Chip) 선택 UI** (SKT / KT / LGU+ 버튼) | 한 클릭으로 선택 |
| 금액 입력 | `type="number"` | **천단위 콤마 포맷** + 금액 프리셋 버튼 | 자릿수 오류 방지 |
| 할인 적용 | 수동 방식/값 입력 | **정책 자동 추천 + 토글** | 할인 미적용/이중적용 방지 |
| 결제수단 | 3개 숫자 필드 | **탭 기반 결제수단 선택** (현금/카드/이체) | 결제 경로 명확화 |
| 색상 | 텍스트 입력 | **미리 등록된 색상 목록에서 선택** | 색상명 불일치 방지 |
| 용량 | 텍스트 입력 | **버튼 선택** (128GB / 256GB / 512GB / 1TB) | 표기 방식 통일 |

### 13-2. 검색형 Combobox 상세 설계

**고객 선택 Combobox**:
```
┌──────────────────────────────┐
│ 🔍 고객 검색 (이름/연락처)     │  ← 텍스트 입력 시작 즉시 필터
├──────────────────────────────┤
│ 김민수  010-1234-5678  SKT   │  ← 이름 + 연락처 + 현재통신사
│ 이지현  010-9876-5432  KT    │
│ 박서준  010-5555-1234  LGU+  │
├──────────────────────────────┤
│ ＋ 신규 고객 등록             │  ← 검색 결과 없을 때 바로 등록
└──────────────────────────────┘
```

**핵심 포인트**:
- 2글자 이상 입력 시 자동 필터링 (디바운스 300ms)
- 연락처 뒤 4자리로도 검색 가능 (매장에서 흔한 패턴)
- 결과 없으면 **"신규 고객 등록" 버튼 인라인 노출** → 페이지 이동 없이 등록

**재고 선택 Combobox**:
```
┌──────────────────────────────────────────┐
│ 🔍 재고 검색 (IMEI, 기기명)                │
├──────────────────────────────────────────┤
│ SKT  Galaxy S25 Ultra  Titanium Gray     │
│      256GB  IMEI 3567...678  ₩912,000    │
│ ─────────────────────────────────────── │
│ KT   iPhone 16 Pro  Desert Titanium      │
│      512GB  IMEI 4901...234  ₩1,150,000  │
└──────────────────────────────────────────┘
```

**핵심 포인트**:
- IMEI 뒤 4자리로도 검색 가능
- 각 항목에 통신사·기기명·색상·용량·입고가를 **2줄 레이아웃**으로 표시
- 선택 즉시 **통신사, 기기원가, 색상, 용량이 자동 채워짐**

### 13-3. 연쇄 선택 패턴 (Cascading Selection)

판매 등록 시 선택 항목 간의 **의존 관계를 UI로 강제**합니다:

```
재고 선택 ──→ 통신사 자동결정 ──→ 요금제 목록 필터 ──→ 부가서비스 필터
    │                                    │
    ├──→ 기기명 자동결정 ──→ 리베이트 정책 자동매칭
    │                                    │
    └──→ 기기원가 자동결정 ──→ 수익 계산 자동시작
```

**현재 문제**: 재고를 선택하면 통신사가 암묵적으로 결정되지만, 사용자가 이를 명확히 인식하기 어려움

**개선**: 선택 흐름을 **시각적 스텝**으로 보여주기

```
Step 1 ──[완료]── Step 2 ──[완료]── Step 3 ──[완료]── Step 4 ──[대기]──
고객 선택          재고 선택          요금제·할인         결제·수납
김민수 ✓          S25 Ultra ✓       5G 프리미엄 ✓       입력 대기 중
```

### 13-4. 금액 입력 오기입 방지

#### 천단위 콤마 포맷터

```
현재: <input type="number" placeholder="1288000" />
     → 사용자가 128만8천을 입력할 때 자릿수 실수 가능

개선: <input placeholder="1,288,000" />
     → 입력 중 실시간으로 콤마 포맷팅
     → 내부적으로는 숫자로 저장
```

#### 금액 프리셋 버튼

자주 사용하는 금액을 버튼으로 빠르게 선택:

```
단말 판매가:
┌────────────────────────────────────────┐
│  [990,000] [1,100,000] [1,288,000]     │  ← 자주 쓰는 금액 프리셋
│  [직접 입력: ____________]              │  ← 프리셋에 없으면 수동 입력
└────────────────────────────────────────┘

수납 금액:
┌────────────────────────────────────────┐
│  [전액 수납 ₩1,288,000]               │  ← 가장 흔한 케이스 원클릭
│  [직접 입력: ____________]              │
└────────────────────────────────────────┘
```

#### 금액 교차 검증 안내

```
┌─ 금액 검증 결과 ──────────────────────┐
│ ✅ 정가 ₩1,288,000                    │
│ ✅ 할인 -₩128,800 (10%)               │
│ ✅ 최종판매가 ₩1,159,200              │
│ ✅ 현금수납 ₩500,000 + 카드 ₩500,000  │
│ ⚠️ 미수금 ₩159,200 발생 예정          │  ← 노란색 경고로 주의 환기
│ ✅ 리베이트 ₩50,000 (정책 자동매칭)    │
```

## 14. 판매 등록 위저드 설계

### 14-1. 현재 판매 등록 폼 문제점

현재 모든 필드가 **한 화면에 펼쳐져** 있음 (2열 레이아웃, ~700줄 컴포넌트):
- 고객 선택, 재고 선택, 요금제, 할인, 리베이트, 수납, 부가서비스, 메모가 동시에 노출
- 사용자가 **어디부터 채워야 할지** 혼란 가능
- 선택 순서를 강제하지 않으므로 **벗어난 순서로 입력 시 오류 가능**

### 14-2. 위저드 단계 설계

판매 등록을 **4단계 위저드**로 분리합니다. 각 단계를 완료해야 다음으로 넘어감:

```
Step 1: 고객·기기    Step 2: 요금제·서비스    Step 3: 금액·할인    Step 4: 결제·확인
━━━━━━━━━━━━━━━━   ━━━━━━━━━━━━━━━━━━━   ━━━━━━━━━━━━━━━━   ━━━━━━━━━━━━━━━━
```

#### Step 1: 고객·기기 선택

```
┌─────────────────────────────────────────────┐
│  Step 1 / 4  고객과 기기를 선택하세요         │
│                                              │
│  고객 ──────────────────────────────────     │
│  🔍 [고객 검색 (이름/연락처)]                 │
│  → 김민수 / 010-1234-5678 / SKT  ✓          │
│  [＋ 신규 고객 등록]                          │
│                                              │
│  판매 재고 ──────────────────────────────     │
│  🔍 [IMEI 또는 기기명 검색]                   │
│  → SKT Galaxy S25 Ultra                      │
│    Titanium Gray / 256GB / ₩912,000  ✓       │
│                                              │
│  판매일: [2026-04-14]   담당자: 관리자        │
│                                              │
│              [다음 →]                         │
└─────────────────────────────────────────────┘
```

**이 단계에서 자동결정되는 것**:
- 통신사 (재고의 통신사)
- 기기원가 (재고의 입고가)
- 기기명, 색상, 용량

#### Step 2: 요금제·부가서비스

```
┌─────────────────────────────────────────────┐
│  Step 2 / 4  요금제와 부가서비스를 선택하세요  │
│                                              │
│  선택된 통신사: SKT  (재고 기준 자동결정)      │
│                                              │
│  요금제 ──────────────────────────────────    │
│  ● 5G 프리미엄      월 ₩89,000               │
│  ○ 5G 스탠다드      월 ₩69,000               │
│  ○ 5G 라이트        월 ₩49,000               │
│                                              │
│  부가서비스 ─────────────────────────────     │
│  ☑ 통화녹음 서비스    월 ₩1,100               │
│  ☐ 안심 보험          월 ₩9,900               │
│  ☐ 클라우드 저장소    월 ₩2,200               │
│                                              │
│         [← 이전]  [다음 →]                    │
└─────────────────────────────────────────────┘
```

**이 단계의 설계 포인트**:
- 요금제는 **라디오 버튼 카드** → 한번 클릭으로 선택 (드롭다운 불필요)
- 통신사에 맞는 요금제만 자동 필터링됨 (이미 구현된 로직 활용)
- 부가서비스는 **체크박스 카드** → 다중 선택

#### Step 3: 금액·할인

```
┌─────────────────────────────────────────────┐
│  Step 3 / 4  금액과 할인을 확인하세요          │
│                                              │
│  단말 판매가 ─────────────────────────────    │
│  [990,000] [1,100,000] [●1,288,000]          │  ← 프리셋 버튼
│  또는 직접입력: [____________]                 │
│                                              │
│  할인 ──────────────────────────────────     │
│  ● 정책 할인 적용 (자동)                      │
│    → SKT 봄 프로모션 10% = -₩128,800         │
│  ○ 수동 할인 입력                             │
│  ○ 할인 미적용                                │
│                                              │
│  지원금: [___________]  (0이면 비워두기)       │
│                                              │
│  ┌─ 실시간 계산 결과 ──────────────────┐      │
│  │ 정가         ₩1,288,000            │      │
│  │ 할인         -₩128,800 (10%)       │      │
│  │ 지원금       -₩0                    │      │
│  │ ─────────────────────────          │      │
│  │ 최종판매가   ₩1,159,200  ← 굵게    │      │
│  │ 리베이트     ₩50,000 (정책 자동)    │      │
│  │ 정책수익     ₩35,000               │      │
│  │ 예상총이익   ₩85,000    ← 강조     │      │
│  └────────────────────────────────────┘      │
│                                              │
│         [← 이전]  [다음 →]                    │
└─────────────────────────────────────────────┘
```

#### Step 4: 결제·최종 확인

```
┌─────────────────────────────────────────────┐
│  Step 4 / 4  결제 수단을 입력하고 확인하세요   │
│                                              │
│  결제수단 ──────────────────────────────     │
│  [●전액 수납] [부분 수납] [미수금 전환]        │
│                                              │
│  금액 입력 (전액 수납 ₩1,159,200 기준):       │
│  [● 현금]  [ 카드 ]  [ 계좌이체 ]             │
│  금액: [1,159,200]                            │
│  카드할부: [일시불▾]                           │
│                                              │
│  ┌─ 최종 확인 ─────────────────────────┐     │
│  │ 고객    김민수 (010-1234-5678)      │     │

│  └─────────────────────────────────────┘     │
│                                              │

```
