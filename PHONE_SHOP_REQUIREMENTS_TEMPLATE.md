# PhoneShop Requirements Template

문서 허브: [PROJECT_DOCS.md](./PROJECT_DOCS.md)

이 문서는 휴대폰 매장에서 사용할 장부형 웹사이트 요구사항을 정리하기 위한 입력 템플릿입니다.

작성 원칙:
- 기능 하나당 하나의 블록으로 작성
- "무엇을", "누가", "언제", "어떤 규칙으로" 쓰는지 분리해서 작성
- 아직 확정되지 않은 내용은 `미정`으로 표시
- 중요도는 `높음 / 보통 / 낮음` 중 하나로 작성

---

## 1. 프로젝트 기본 정보

### 1-1. 프로젝트 한줄 설명
- 입력: 휴대폰 매장에서 판매, 수납, 미수금, 재고를 관리하는 장부 웹사이트

### 1-2. 이 웹사이트를 사용하는 사람
- 입력: 직원, 관리자

### 1-3. 가장 중요한 목표 3개
1. 판매 등록을 빠르게 끝내고, 장부 누락 없이 저장할 수 있어야 한다.
2. 재고, 판매, 미수금, 수납이 한 흐름으로 연결되어야 한다.
3. 리베이트, 기기값, 부가세 관련 금액을 거래 기준으로 추적할 수 있어야 한다.

### 1-4. 가장 먼저 필요한 메뉴
- 재고 관리
- 요금제 관리
- 고객 관리
- 판매 관리
- 정책 관리
- 대시 보드
- 일정 관리
- 직원 관리
- 미수금 관리

### 1-5. 주 사용 환경
- 입력: 매장 PC 중심 / 태블릿도 사용 / 모바일은 조회만

### 1-6. 로그인 권한 구분 필요 여부
- 입력: 필요

### 1-7. 외부 연동 또는 부가 기능
- 입력: 엑셀 다운로드, 문자 발송, 프린트, 파일 첨부, 카카오 알림톡

### 1-8. 지점 운영 방식
- 입력: 1차 버전은 단일 매장 기준으로 시작하고, 추후 다지점 확장 가능하게 설계

### 1-9. 금액 처리 기준
- 입력: 1차 버전은 매장 장부 기준으로 총액 입력 중심으로 운영하고, 판매건마다 기기원가 / 적용 할인 / 적용 리베이트 / 정책 수익 / 부가세를 별도 기록한다. 정식 회계 마감 기능은 2차 개발로 미룬다.

### 1-10. 리베이트 정의
- 입력: 리베이트는 고객 할인 항목이 아니라, 판매 시 `통신사 + 기기모델 + 판매일` 조건에 따라 공급업체가 매장에 지급하는 추가 수익이다. 판매 시 실제 적용 금액을 입력하며, 0원 입력도 허용한다.

### 1-11. 수익 계산 기준
- 입력: 판매건 수익은 `적용 리베이트 + 정책 수익 - 차감 항목` 기준으로 본다. 할인은 고객 판매금액에 영향을 주는 항목이고, 리베이트/정책 수익은 매장 수익 항목으로 별도 기록한다. 1차 버전에서는 차감 항목이 없으면 0으로 처리하고, 정책 수익은 통신사별 정책에 따라 자동 계산하거나 0으로 둘 수 있다.

## 2. 사용자 권한 정리

### 2-1. 관리자
- 할 수 있는 일: 모든 기능 사용 가능

### 2-2. 직원
- 할 수 있는 일: 재고, 고객, 판매, 미수금 조회/등록 가능. 정책 관리, 직원 관리, 삭제 기능은 제한.

### 2-3. 기타 권한이 있으면 추가
- 권한명: 없음(1차 버전은 관리자 / 직원 2권한만 운영 추천)
- 할 수 있는 일: 추후 필요 시 회계/정산 전용 권한 추가

## 3. 전체 메뉴 구조 초안

- `정책 관리`는 직원 인센티브가 아니라 `리베이트 정책`, `판매금액 기반 수익 정책`, `할인 정책` 중심으로 설계하는 것을 추천

- 부가서비스 관리: 상단에 필터(통신사), 부가서비스등록(통신사, 부가서비스명, 부가서비스, 비고)
- 고객 관리: 상단에 필터(통신사, 고객명), 고객등록(통신사, 고객명, 연락처, 생년월일, 주소, 메모)
- 판매 관리: 상단에 필터(통신사, 고객명, 요금제), 판매등록(통신사, 고객명, 요금제, 기기명, 색상, 용량, IMEI, 입고가, 담당 직원, 정가, 할인적용여부, 할인방식(비율/금액), 할인값, 현금, 카드(할부기능))
- 직원 관리: 상단에 필터(지점명, 직원명, 직위), 직원등록(지점명, 직원명, 직위, 연락처, 생년월일, 주소, 메모, 급여, 인센티브(%)), 직원테이블(지점명, 직원명, 직위, 연락처, 생년월일, 주소, 메모, 급여, 인센티브(%))
- 정책 관리: 상단에 필터(통신사, 기기명, 적용일, 정책상태), 리베이트정책등록(적용일, 통신사, 기기명, 기본 리베이트 금액, 비고), 판매수익정책등록(통신사, 정책명, 적용기간, 계산방식(없음/정액/비율), 계산값(0 허용), 비고), 할인정책등록(정책구분(통신사별 할인율/단말기별 할인), 통신사, 기기명, 적용기간, 할인방식(비율/금액), 할인값, 0 허용, 비고), 정책테이블(통신사, 정책명, 적용기간, 상태, 계산방식, 계산값)

- 대시 보드: 추후 항목 지정
- 보고서: 추후 항목 지정

추천 정리:

## 4. 자주 사용하는 업무 흐름

### 흐름 1
- 이름: 신규 고객 휴대폰 판매
- 시작 상황: 고객이 매장에 방문하여 휴대폰 구매를 원함
- 처리 순서:
1. 고객 정보 입력 및 등록
2. 통신사 선택
3. 요금제 선택
4. 기기 선택
5. 판매 정보 입력(미수금 발생 여부 선택)
6. 판매 완료
- 완료 결과: 고객에게 휴대폰 판매 완료, 할인 적용 결과와 최종 판매금액이 저장되고, 미수금 발생 시 미수금 등록, 재고 감소, 고객 판매금액과 리베이트/정책 수익이 함께 기록됨

### 흐름 2
- 이름: 기존 고객 휴대폰 판매
- 시작 상황: 고객이 매장에 방문하여 휴대폰 구매를 원함
- 처리 순서:
1. 고객 정보 검색
2. 통신사 선택
3. 요금제 선택
4. 기기 선택
5. 판매 정보 입력(미수금 발생 여부 선택)
6. 판매 완료
- 완료 결과: 고객에게 휴대폰 판매 완료, 할인 적용 결과와 최종 판매금액이 저장되고, 미수금 발생 시 미수금 등록, 재고 감소, 고객 판매금액과 리베이트/정책 수익이 함께 기록됨

### 흐름 3
- 이름: 기존 고객 휴대폰 교체
- 시작 상황: 고객이 매장에 방문하여 휴대폰 교체를 원함
- 처리 순서:
1. 고객 정보 검색
2. 통신사 선택
3. 요금제 선택
4. 기기 선택
5. 판매 정보 입력
6. 판매 완료
7. 중고기기 반납
8. 중고기기 판매
9. 중고기기 판매 정보 입력 완료
- 완료 결과: 고객에게 휴대폰 판매 완료, 재고 감소, 중고기기 판매 차익과 판매건 수익이 함께 기록됨

### 흐름 4
- 이름: 미수금 수납
- 시작 상황: 고객이 매장에 방문하여 미수금을 수납
- 처리 순서:
1. 고객 정보 검색
2. 미수금 검색
3. 미수금 수납
4. 미수금 수납 완료
- 완료 결과: 고객에게 미수금 수납 완료, 미수금 감소

## 5. 데이터 항목 정리

### 고객
- 이름: 필수, 실명 기준
- 연락처: 필수, 고객 중복 확인 기준
- 생년월일: 선택, 본인 확인용
- 주소: 선택
- 메모: 선택, 상담/특이사항 기록
- 현재 통신사: 선택, 기존 고객 조회 편의용

### 판매
- 판매일: 필수
- 고객: 필수, 기존 고객 선택 또는 신규 등록 연동
- 기기명: 필수
- 통신사: 필수
- 요금제: 필수
- 정가: 필수, 할인 전 기준 판매 금액
- 할인적용여부: 필수, 예 / 아니오
- 할인방식: 선택, 비율 / 금액
- 할인값: 선택, 0 이상
- 할인금액: 자동 계산
- 할인적용 판매가: 자동 계산, 할인 반영 후 고객 청구 금액
- 판매가: 필수, 고객에게 청구하는 최종 판매 금액
- 지원금: 선택, 고객 할인 또는 공시지원/추가지원 포함. 리베이트와는 별도 개념
- 실수납액: 필수, 당일 실제 받은 금액
- 미수금: 자동 계산, `판매가 - 실수납액`
- 담당 직원: 필수
- IMEI: 필수, 판매 재고와 연결
- 기기원가: 필수, 재고 입고가와 연결
- 결제수단: 필수, 현금 / 카드 / 계좌이체 / 혼합
- 카드금액: 선택
- 카드할부개월: 카드 결제 시 필수
- 적용 리베이트: 필수, 판매 시점 실제 금액 입력, 0 허용
- 정책 수익: 자동 계산 또는 수기 보정, 0 허용
- 적용 정책명: 선택, 어떤 정책이 적용되었는지 저장
- 적용 할인 정책명: 선택, 어떤 할인 정책이 적용되었는지 저장
- 수익 계산 기준값: 선택, 정책 계산에 사용한 기준 판매금액 저장
- 총 수익: 자동 계산, `적용 리베이트 + 정책 수익 - 차감 항목`
- 부가세: 선택, 거래 기준 별도 저장

### 수납
- 수납일: 필수
- 관련 판매건: 필수, 판매건 또는 미수금 건과 연결
- 수납금액: 필수
- 수납수단: 필수, 현금 / 카드 / 계좌이체
- 메모: 선택
- 수납 담당 직원: 자동 기록
- 수납 후 잔액: 자동 계산
- 완납 여부: 자동 계산

### 재고
- 통신사: 필수
- 기기명: 필수
- 색상: 필수
- 용량: 필수
- IMEI 또는 시리얼: 필수, 유일값
- 입고일: 필수
- 출고 상태: 필수, 입고 / 예약 / 판매 / 반품 / 폐기
- 메모: 선택
- 입고가: 필수
- 담당 직원: 선택
- 출고일: 판매 시 자동 기록

### 직원 실적
- 직원명: 기준값
- 기간: 일 / 주 / 월
- 판매 건수: 자동 계산
- 고객 판매금액: 자동 계산
- 리베이트 수익: 자동 계산
- 정책 수익: 자동 계산
- 총 수익: 자동 계산
- 수납 금액: 자동 계산
- 인센티브: 추후 별도 정책 기준 또는 수기 조정

### 정책
- 통신사: 필수
- 정책명: 필수
- 정책구분: 필수, 리베이트 정책 / 판매수익 정책 / 할인 정책
- 적용 시작일: 필수
- 적용 종료일: 선택
- 기기명: 리베이트 정책일 때 필수
- 기본 리베이트 금액: 리베이트 정책일 때 사용, 0 허용
- 계산방식: 판매수익 정책일 때 필수, 없음 / 정액 / 비율
- 계산값: 판매수익 정책일 때 필수, 0 허용
- 할인대상: 할인 정책일 때 필수, 통신사 / 단말기
- 할인방식: 할인 정책일 때 필수, 비율 / 금액
- 할인값: 할인 정책일 때 필수, 0 허용
- 비고: 선택

### 공통 데이터 규칙
- 고객의 기본 식별 기준은 `연락처`로 한다.
- 재고의 기본 식별 기준은 `IMEI`로 한다.
- 고객 1명은 판매건 여러 개를 가질 수 있다.
- 판매 1건은 기본적으로 기기 1대 기준으로 시작한다.
- 미수금은 반드시 판매건과 연결한다.
- 수납은 분할 수납을 허용한다.
- 삭제는 실제 삭제보다 `숨김 처리`를 우선한다.
- 누가 언제 수정했는지 수정 이력을 남긴다.
- 금액은 1차에서 총액 기준으로 입력하고, 필요한 세부 금액은 별도 필드로 저장한다.
- 리베이트는 고객 할인값이 아니라 공급업체가 매장에 지급하는 수익 값으로 저장한다.
- 판매 시점의 실제 리베이트 금액을 판매건에 고정 저장하고, 이후 정책이 바뀌어도 과거 판매건 값은 바뀌지 않는다.
- 판매금액 기반 수익 계산 로직은 통신사별 정책으로 관리한다.
- 정책 계산값이 0이면 해당 판매건은 판매금액 기반 수익을 반영하지 않는다.
- 지원금과 리베이트는 반드시 별도 필드로 분리한다.
- 할인은 리베이트와 별도 필드로 분리한다.
- 할인 정책은 `통신사별 할인율` 또는 `단말기별 할인`으로 관리한다.
- 판매 시 할인 적용 여부를 직접 선택할 수 있다.
- 할인 적용 시 `비율`과 `금액` 중 하나를 선택해 입력할 수 있다.
- 정책 할인값이 있어도 판매 시 0 또는 미적용으로 변경할 수 있다.
- 할인 정책 우선순위는 `단말기별 할인 > 통신사별 할인율`로 적용하고, 판매 시 수기 변경을 허용한다.

- 입력: 관리자, 직원

### 언제 사용하는지
- 입력: 판매 후 미수금 발생 시, 고객 수납 시, 잔액 확인 시

### 입력해야 하는 정보
- 고객 / 관련 판매건
- 미수금 금액 / 수납 금액
- 수납일 / 수납수단 / 메모

### 저장 후 보여줘야 하는 결과
- 미수금 잔액 실시간 반영
- 완납 여부 자동 표시
- 고객 상세와 판매 상세에 수납 이력 표시

### 꼭 지켜야 하는 규칙
- 수납 금액은 현재 미수금 잔액보다 클 수 없음
- 분할 수납 허용
- 완납 시 자동으로 상태를 완료로 변경

### 수정 / 삭제 규칙
- 수납 내역은 삭제 대신 취소 처리
- 관리자만 금액 조정 가능, 조정 사유 필수

### 검색 / 필터 조건
- 고객명
- 기간
- 미수금 상태(미납 / 부분수납 / 완납)

### 예외 상황
- 초과 수납 시도
- 판매 취소 후 이미 일부 수납된 경우

### 화면에서 중요한 점
- 입력: 현재 잔액과 수납 후 잔액이 크게 보여야 하고, 고객 검색이 빨라야 함

### 중요도
- 입력: 높음

### 완료 기준
- 판매에서 미수금 자동 생성 가능
- 수납 등록 가능
- 고객별 잔액 및 수납 이력 조회 가능

## 7. 보고서 / 통계 요구사항

### 꼭 보고 싶은 숫자
- 입력: 오늘 판매 건수, 오늘 수납 금액, 이번 달 고객 판매금액, 이번 달 적용 리베이트 합계, 이번 달 정책 수익 합계, 이번 달 총 수익, 미수금 합계, 완납/미납 건수, 직원별 판매 실적

### 기간별 조회가 필요한지
- 입력: 일간 / 주간 / 월간 / 기간 직접 선택

### 다운로드가 필요한 형식
- 입력: 엑셀 / PDF / 프린트

## 8. 보안 / 운영 정책

### 접근 제한
- 입력: 로그인한 사람만 접근 가능, 권한별 메뉴 제한 적용

### 기록 보존
- 입력: 삭제 대신 숨김 처리, 수정 이력 남기기

### 백업 또는 데이터 안정성 요구
- 입력: 매일 자동 백업, 장애 시 최근 백업으로 복구 가능해야 함

## 9. 1차 개발 범위

### 이번에 꼭 만들 기능
1. 로그인 및 권한 관리
2. 기초정보 관리(통신사 / 요금제 / 부가서비스 / 정책)
3. 재고 관리
4. 고객 관리
5. 판매 관리 + 미수금 관리

### 있으면 좋은 기능
- 직원 실적 화면
- 대시보드 요약 화면
- 엑셀 / PDF 출력

### 나중에 추가해도 되는 기능
- 일정 관리
- 문자 발송 / 카카오 알림톡
- 파일 첨부 / 고급 프린트 양식

## 10. 참고사항 / 메모

- 참고하고 싶은 기존 장부 방식: 엑셀 장부 또는 수기 장부 기준과 최대한 비슷한 입력 흐름 추천
- 현재 오프라인에서 쓰는 방식: 미정, 실제 사용 중인 양식이 있으면 추후 반영
- 꼭 비슷하게 만들고 싶은 사이트 또는 프로그램: 미정
- 기타 메모: 1차 목표는 복잡한 디자인보다 빠른 입력 / 빠른 조회 / 정확한 금액 기록
